Start with the mechanism. When a transformer generates text, every new token attends to every previous one. Recomputing all that attention from scratch for each token would be quadratic madness, so inference engines cache the key and value projections of everything already processed. That cache is the KV cache, and it grows linearly with context length — every layer, every attention head, every token, kept resident in GPU memory for as long as your conversation lives.

Now the consequence, because every section has to earn its place with one. A long chat with a large model can drag gigabytes of cache behind it, and that memory is rented from the same pool that holds the model weights. When the cache fills the card, your provider either evicts you, shards you, or charges you. This is why long context is priced the way it is, why prefill and decode feel like two different products, and why prompt caching discounts exist: reusing a cached prefix is the one free lunch in inference.

The engineering pattern that falls out is simple to state: structure your prompts so the stable part comes first and the variable part comes last. A system prompt that never changes is cache gold; a timestamp in the first line is cache poison. Teams that internalize this see latency and cost drop without touching a single model parameter.

Three takeaways. The KV cache converts context length into memory pressure, linearly. Memory pressure, not compute, sets the real price of long conversations. And prompt structure is a performance knob you already own. Next on the curriculum: sampling — temperature, top-p, and why the model is less random than you think.
